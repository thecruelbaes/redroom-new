# Деплой на VPS — пошагово

Цель: поднять сайт на новом VPS и посадить на домен **nvrskmusic.ru**.
Подробный бизнес-контекст (Метрика, Директ, редирект и т.д.) — в `../RedRoom_Studio_план.md`, раздел 7.
Здесь — только техническая часть: сервер → Docker → nginx → SSL.

## Архитектура: сайт + в будущем CRM + бот на одном сервере

На этом же VPS позже появятся CRM и, возможно, бот с уведомлениями. Чтобы они не мешали
друг другу ни сейчас, ни потом:

- **Один сервис = своя папка = свой `docker-compose.yml`.** Обновление/рестарт одного
  сервиса (`docker compose up -d --build` внутри его папки) никогда не трогает соседние —
  у каждого свой независимый жизненный цикл, свой образ, свои логи.
- **Общая внешняя Docker-сеть `redroom-net`** — создаётся один раз на сервере, каждый
  сервис в своём compose-файле просто подключается к ней (`external: true`). Это даёт
  сервисам возможность видеть друг друга по имени контейнера при необходимости (например,
  бот вызывает внутренний API CRM), не открывая порты наружу.
- **Наружу торчит только nginx.** Все контейнеры биндятся на `127.0.0.1:<порт>`, а не
  `0.0.0.0` — снаружи их портов просто не существует, только через nginx по нужному
  (под)домену.
- **Резерв портов** (чтобы не столкнуться при добавлении новых сервисов):

  | Порт | Сервис | Статус |
  |------|--------|--------|
  | 3000 | Сайт (redroom-site) | занят |
  | 3001 | CRM | зарезервирован |
  | 3002 | Бот (если будет свой HTTP/webhook) | зарезервирован |

- **Один (под)домен = один файл в `nginx/sites-available/`.** Сайт — `nvrskmusic.ru`.
  Когда появится CRM — скорее всего свой поддомен (`crm.nvrskmusic.ru`), отдельный файл,
  отдельный `certbot` для SSL. Общий nginx.conf никто не трогает.
- **Папки на сервере** — единый родитель для всех сервисов, а не вперемешку в `~`:
  ```
  ~/apps/
    ├── redroom-new/     ← этот репозиторий (сайт)
    ├── redroom-crm/     ← будущее
    └── redroom-bot/     ← будущее
  ```

⚠️ Пока не решён стек CRM (Node? своя БД или тот же Supabase?) — этот план описывает
только *форму* (изоляция, сеть, порты), не может заранее учесть конкретные нужды CRM
(например, свой контейнер с Postgres и его бэкапы). Уточнить, когда стек определится.

## Что понадобится
- VPS с Ubuntu 22.04/24.04 (или аналог), доступ по SSH.
- Домен `nvrskmusic.ru` — доступ к DNS-записям у регистратора.
- Реальные значения `.env` (см. ниже) — токен Telegram-бота для формы заявок.

## 1. Подготовка сервера (один раз для всех будущих сервисов)

```bash
apt update && apt upgrade -y
apt install -y docker.io docker-compose-plugin nginx certbot python3-certbot-nginx git
systemctl enable --now docker

# Общая сеть для всех сервисов (сайт, потом CRM, потом бот) — один раз навсегда
docker network create redroom-net

mkdir -p ~/apps
```

`docker-compose-plugin` — отдельно, `docker.io` его не тянет по умолчанию на Ubuntu, а без
него команда `docker compose` ниже просто не найдётся.

## 2. DNS

У регистратора `nvrskmusic.ru` создать A-запись, указывающую на IP нового VPS:
```
A   @     <IP_VPS>
A   www   <IP_VPS>
```
Подождать распространения DNS (обычно от нескольких минут до пары часов).

⚠️ **DNS на новый сервер переключаем в последнюю очередь** — только после того, как сайт
проверен по IP/hosts-файлу и закрыт чек-лист по Метрике/Директу (цель `lead_submit`,
ссылки в объявлениях, верификация в Вебмастере). До этого момента старый сайт продолжает
работать на текущем хостинге как обычно.

## 3. Код на сервер

```bash
cd ~/apps
git clone https://github.com/thecruelbaes/redroom-new.git
cd redroom-new
```

## 4. `.env` на сервере

Создать `.env` в корне проекта (значения — реальные, не коммитить):
```
TELEGRAM_BOT_TOKEN=...
TELEGRAM_CHAT_ID=...
NEXT_PUBLIC_SITE_URL=https://nvrskmusic.ru
NEXT_PUBLIC_YANDEX_VERIFICATION=...   # если нужна верификация в Яндекс.Вебмастере

# Без этих двух /admin не запустится (сборка падает при старте, см. lib/auth.ts) —
# это специально, чтобы админка не оказалась защищена дефолтным паролем из репозитория.
ADMIN_PASSWORD=...
ADMIN_SESSION_SECRET=...

# Supabase — без них отзывы работают на статичном фоллбеке из кода, админка не пишет в БД.
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
SUPABASE_SERVICE_ROLE_KEY=...
```

## 5. Собрать и запустить контейнер

```bash
docker compose up -d --build
```

Проверить: `curl -I http://127.0.0.1:3000` должен вернуть `200`. Контейнер слушает только
`127.0.0.1` (см. `docker-compose.yml`) — снаружи сервера порт 3000 недоступен, это ожидаемо
и правильно, доступ наружу будет только через nginx (следующий шаг).

## 6. nginx — реверс-прокси

```bash
cp deploy/nginx.conf.example /etc/nginx/sites-available/nvrskmusic.ru
ln -s /etc/nginx/sites-available/nvrskmusic.ru /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx
```

Проверить: `http://nvrskmusic.ru` должен открыть сайт (пока без HTTPS).

## 7. SSL (HTTPS)

```bash
certbot --nginx -d nvrskmusic.ru -d www.nvrskmusic.ru
```
Certbot сам допишет конфиг nginx (редирект 80→443, сертификат) и настроит автопродление.

## 8. Финальная проверка
- [ ] `https://nvrskmusic.ru` открывается, есть замочек (валидный SSL).
- [ ] Форма на сайте реально долетает до Telegram (`/api/lead`).
- [ ] Открывается с мобильного интернета (не только с сервера/офиса).
- [ ] `/admin` открывается и пускает по реальному паролю (не по дефолтному — см. `lib/auth.ts`).

## Обновление сайта после изменений (redeploy)

```bash
cd ~/apps/redroom-new
git pull
docker compose up -d --build
```
Это пересобирает и перезапускает **только** контейнер сайта — CRM/бот (когда появятся, в
своих папках) это никак не затронет.

## Когда добавится CRM или бот

1. Клонировать в `~/apps/redroom-crm` (или `redroom-bot`), со своим `docker-compose.yml` —
   скопировать паттерн из этого репозитория: `container_name`, `env_file`, порт из таблицы
   резерва выше, `127.0.0.1:<порт>:<порт>`, подключение к `networks: redroom-net (external: true)`.
2. Свой файл в `/etc/nginx/sites-available/` под нужный (под)домен, `nginx -t && systemctl reload nginx`.
3. Свой `certbot --nginx -d ...` для SSL этого поддомена.
4. Если сервису нужна своя БД (например, Postgres для CRM) — отдельный контейнер в том же
   compose-файле сервиса, **плюс сразу продумать бэкап** (это тогда единственные данные,
   которых нет во внешнем Supabase) — не откладывать на потом.
